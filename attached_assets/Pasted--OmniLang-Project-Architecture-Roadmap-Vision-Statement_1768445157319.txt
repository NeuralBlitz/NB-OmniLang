# OmniLang: Project Architecture & Roadmap

## 🎯 Vision Statement

**Transform Markdown from a static documentation format into a dynamic, computational platform that unifies data, logic, and visualization.**

OmniLang bridges the gap between:

- Static documents (Markdown, PDF)
- Computational notebooks (Jupyter, Observable)
- Data formats (JSON, YAML, TOML)
- Modern web applications

## 📐 Core Architecture

### System Design

```
┌─────────────────────────────────────────────────────────┐
│                  OmniLang Ecosystem                      │
└─────────────────────────────────────────────────────────┘
                           │
        ┌──────────────────┼──────────────────┐

```

### Component Breakdown

**1. Core Engine (`omni-lang.js`)**

- Fence parser (regex-based)
- AST builder
- Execution engine
- Scope management
- Dependency analyzer (basic)
- Helper function library

**2. CLI Tool (`omni-cli.js`)**

- Build command (compile to HTML)
- Watch command (auto-rebuild)
- Validate command (syntax check)
- File I/O handling

**3. Renderers**

- Markdown → HTML converter
- Chart.js integration
- Template engine (basic)

## 🔧 Current Implementation (v0.1)

### What Works ✅

**Fence Types:**

- `omni:data` - JSON data loading
- `omni:compute` - JavaScript execution
- `omni:chart` - Chart.js visualizations

**Features:**

- Named data blocks
- Named compute blocks
- Inline expression evaluation
- Static HTML compilation
- Built-in helper functions
- CLI with build/watch/validate

**Helper Functions:**

- `len()`, `sum()`, `avg()`, `max()`, `min()`
- `filter()`, `map()`, `groupBy()`
- Access to Math, JSON, etc.

### Known Limitations ⚠️

1. **Dependency Order** - Sequential execution means charts/compute referencing other computed values can fail if order is wrong
1. **Inline Fence Type** - Currently shows as “unknown type” but still works
1. **Error Handling** - Basic, could be more graceful
1. **Data Formats** - JSON only (no YAML/TOML/CSV yet)
1. **Output Formats** - HTML only (no PDF/DOCX yet)

## 🏗️ Technical Decisions

### Language Choice

**Current: JavaScript/Node.js**
- ✅ Easy distribution
- ✅ Browser compatibility
- ✅ Rich ecosystem
- ❌ Less performant than compiled languages

**Future Considerations:**
- Rust core for performance
- WASM compilation for browser
- Language bindings (Python, Ruby)

### Parser Strategy

**Current: Regex-based**
- ✅ Simple, fast for basic cases
- ✅ Easy to extend
- ❌ Limited error messages
- ❌ Fragile for complex nesting

**Future: Proper Parser**
- PEG grammar (parsimmon, peggy)
- Better error recovery
- Source maps
- AST transformations

### Execution Model

**Current: Synchronous interpretation**
- ✅ Simple to implement
- ❌ Sequential only
- ❌ No optimization

**Future: Smart execution**
- Topological sorting
- Parallel where possible
- Memoization/caching
- JIT for hot paths

## 📊 Comparison with Alternatives

| Feature | OmniLang | Jupyter | R Markdown | MDX | Observable |
|---------|----------|---------|------------|-----|------------|

| No server needed | ✅ | ❌ | ❌ | Depends | ❌ |
| Multiple languages | 🔜 | ✅ | ✅ | ❌ | ✅ |
| Static export | ✅ | ✅ | ✅ | ✅ | ✅ |
| Interactive UI | 🔜 | Limited | Limited | ✅ | ✅ |
| Data viz | ✅ | ✅ | ✅ | Depends | ✅ |
| Version control friendly | ✅ | ❌ | ✅ | ✅ | ❌ |

**OmniLang's Niche:**
- Simpler than Jupyter (just Markdown!)
- More powerful than MDX (computation!)
- More portable than Observable (static HTML!)
- More flexible than R Markdown (any language!)
